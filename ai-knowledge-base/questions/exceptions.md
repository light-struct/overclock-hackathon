### Q1: Что такое исключение (exception) в Java?

- **ID:** `exc-easy-01`
- **Difficulty:** easy
- **Tags:** исключения, основы

**Варианты:**
- A) Тип данных для хранения кодов ошибок
- B) Предупреждение компилятора о потенциальной проблеме
- C) Событие, нарушающее нормальное выполнение программы, представленное объектом
- D) Логическая ошибка в алгоритме

**Ответ:** `C`

**Объяснение:** Исключение — это объект, представляющий аномальное событие во время выполнения программы. Когда возникает ошибка, JVM создаёт объект исключения и "выбрасывает" его. Этот объект содержит информацию об ошибке: тип, сообщение, стек вызовов.

### Q2: Что является базовым классом для всех исключений и ошибок в Java?

- **ID:** `exc-easy-02`
- **Difficulty:** easy
- **Tags:** иерархия-исключений, Throwable

**Варианты:**
- A) Exception
- B) RuntimeException
- C) Error
- D) Throwable

**Ответ:** `D`

**Объяснение:** Иерархия: `Throwable` → `Error` (серьёзные проблемы JVM: OutOfMemoryError, StackOverflowError) и `Exception` → `RuntimeException` (unchecked) + остальные (checked). `Throwable` — единственный класс, который можно бросить через `throw` и поймать в `catch`.

### Q3: Что делает ключевое слово throw?

- **ID:** `exc-easy-03`
- **Difficulty:** easy
- **Tags:** throw, исключения

**Варианты:**
- A) Перехватывает исключение в блоке catch
- B) Объявляет, что метод может выбросить исключение
- C) Явно выбрасывает объект исключения
- D) Завершает выполнение метода и возвращает управление

**Ответ:** `C`

**Объяснение:** `throw new SomeException("message")` — явно выбрасывает исключение. Выполнение текущего метода прерывается, JVM ищет подходящий `catch` вверх по стеку вызовов. Можно бросать только объекты типа `Throwable` или его наследников.

### Q4: Что делает ключевое слово throws в сигнатуре метода?

- **ID:** `exc-easy-04`
- **Difficulty:** easy
- **Tags:** throws, checked-exceptions

**Варианты:**
- A) Немедленно выбрасывает указанное исключение
- B) Объявляет, что метод может выбросить указанное checked-исключение
- C) Делает метод обязательным для переопределения в подклассе
- D) Принудительно оборачивает тело метода в try-catch

**Ответ:** `B`

**Объяснение:** `throws` в сигнатуре — декларация: "этот метод может выбросить данное исключение, вызывающий код должен его обработать или передать дальше". Обязателен для checked-исключений. Для unchecked (RuntimeException) — опционален.

### Q5: Что делает блок finally?

- **ID:** `exc-easy-05`
- **Difficulty:** easy
- **Tags:** finally, try-catch

**Варианты:**
- A) Выполняется только если исключений не было
- B) Выполняется только при возникновении исключения
- C) Завершает программу после блока try
- D) Выполняется всегда — независимо от того, было ли исключение

**Ответ:** `D`

**Объяснение:** `finally` выполняется всегда: и при нормальном выполнении, и при исключении, и при `return` из `try`. Исключение: `System.exit()` или фатальный сбой JVM. Используется для освобождения ресурсов (закрытие файлов, соединений). С Java 7 — лучше использовать try-with-resources.

### Q6: Что такое NullPointerException?

- **ID:** `exc-easy-06`
- **Difficulty:** easy
- **Tags:** NullPointerException, unchecked

**Варианты:**
- A) Checked исключение при попытке создать объект с null-параметром
- B) Unchecked исключение при попытке использовать метод или поле у null-ссылки
- C) Исключение при переполнении стека вызовов
- D) Исключение при выходе за границу массива

**Ответ:** `B`

**Объяснение:** NPE возникает при: вызове метода у null (`null.toString()`), обращении к полю null (`null.field`), работе с null-массивом. NPE — наследник RuntimeException (unchecked). С Java 14 появились "helpful NullPointerExceptions" с подробным сообщением о том, что именно было null.

### Q7: Можно ли иметь несколько блоков catch после одного try?

- **ID:** `exc-easy-07`
- **Difficulty:** easy
- **Tags:** try-catch, иерархия-исключений

**Варианты:**
- A) Нет, только один catch на каждый try
- B) Да, каждый catch перехватывает свой тип исключения
- C) Только если все catch-блоки пустые
- D) Максимум два catch-блока

**Ответ:** `B`

**Объяснение:** После одного `try` можно поставить несколько `catch` блоков. JVM выберет первый подходящий сверху вниз. Важен порядок: сначала более специфичные (дочерние), потом общие (родительские). `catch(Exception e)` перехватит всё, поэтому ставится последним.

### Q8: В чём разница между checked и unchecked исключениями?

- **ID:** `exc-medium-01`
- **Difficulty:** medium
- **Tags:** checked, unchecked, иерархия

**Варианты:**
- A) Checked — наследники RuntimeException; unchecked — наследники Exception
- B) Checked — компилятор требует обработки или объявления через throws; unchecked (RuntimeException и его наследники) — не требует
- C) Checked исключения работают быстрее unchecked
- D) Checked исключения нельзя поймать в catch-блоке

**Ответ:** `B`

**Объяснение:** Checked: наследники Exception (кроме RuntimeException) — `IOException`, `SQLException`. Компилятор требует обработки: либо `try-catch`, либо `throws` в сигнатуре. Unchecked: наследники RuntimeException — `NPE`, `ArrayIndexOutOfBounds`. Обработка не обязательна. Ошибки (Error) тоже unchecked.

### Q9: В каком порядке размещать блоки catch при иерархии исключений?

- **ID:** `exc-medium-02`
- **Difficulty:** medium
- **Tags:** try-catch, порядок, иерархия

**Варианты:**
- A) Сначала базовые (родительские), потом специфичные — произвольный порядок
- B) Сначала специфичные (дочерние) исключения, потом более общие (родительские)
- C) Порядок не важен, JVM сама выберет правильный
- D) По алфавитному порядку имён классов исключений

**Ответ:** `B`

**Объяснение:** Сначала — более специфичные: `catch(FileNotFoundException e)`, потом — более общие: `catch(IOException e)`, последним — самое общее: `catch(Exception e)`. Если поставить `catch(Exception e)` первым — компилятор выдаст ошибку "unreachable code" для следующих catch-блоков, так как Exception перехватит всё.

### Q10: Как создать своё пользовательское исключение в Java?

- **ID:** `exc-medium-03`
- **Difficulty:** medium
- **Tags:** custom-exception, наследование

**Варианты:**
- A) Реализовать интерфейс Throwable
- B) Наследовать класс Exception (checked) или RuntimeException (unchecked)
- C) Использовать аннотацию @Exception над классом
- D) Переопределить метод throw() в своём классе

**Ответ:** `B`

**Объяснение:** `class InsufficientFundsException extends RuntimeException { public InsufficientFundsException(String msg) { super(msg); } }`. Наследуй от RuntimeException для unchecked (не требует обработки), от Exception для checked. Добавляй конструкторы с message и cause.

### Q11: Что такое multi-catch (Java 7+)?

- **ID:** `exc-medium-04`
- **Difficulty:** medium
- **Tags:** multi-catch, java7, синтаксис

**Варианты:**
- A) Цепочка из нескольких отдельных catch-блоков
- B) `catch (ExceptionA | ExceptionB e)` — перехват нескольких типов в одном блоке
- C) Вложенные try-catch блоки
- D) Перехват массива объектов исключений

**Ответ:** `B`

**Объяснение:** Multi-catch (Java 7): `catch (IOException | SQLException e) { handle(e); }` — перехватывает оба типа в одном блоке. Переменная `e` в multi-catch неявно final — нельзя переприсвоить. Типы не должны быть в отношении наследования. Сокращает дублирование кода.

### Q12: Что такое try-with-resources и зачем он нужен?

- **ID:** `exc-medium-05`
- **Difficulty:** medium
- **Tags:** try-with-resources, AutoCloseable, ресурсы

**Варианты:**
- A) Блок try с несколькими вложенными catch-блоками
- B) Конструкция, автоматически закрывающая ресурсы, реализующие AutoCloseable, после блока try
- C) Попытка выполнить код с автоматическим перехватом всех исключений
- D) Синоним для блока finally с ручным закрытием потоков

**Ответ:** `B`

**Объяснение:** `try (FileReader fr = new FileReader(path)) { ... }` — `fr.close()` вызывается автоматически после try. Ресурс должен реализовывать `AutoCloseable` (метод `close()`). Можно открыть несколько ресурсов: `try (A a = ...; B b = ...)`. Закрываются в обратном порядке. Заменяет try-finally для работы с ресурсами.

### Q13: Что такое цепочка исключений (exception chaining)?

- **ID:** `exc-medium-06`
- **Difficulty:** medium
- **Tags:** exception-chaining, cause, wrapping

**Варианты:**
- A) Несколько catch-блоков, обрабатывающих одно исключение по очереди
- B) Обёртывание оригинального исключения в новое с передачей как cause для сохранения контекста
- C) Иерархия классов исключений (наследование)
- D) Упорядоченный список всех выброшенных исключений в сессии

**Ответ:** `B`

**Объяснение:** `throw new ServiceException("DB error", originalException)` — новое исключение содержит причину (cause). `e.getCause()` возвращает оригинальное. Это позволяет: не терять низкоуровневую информацию, оборачивать checked в unchecked, добавлять контекст. Конструктор с двумя параметрами: `Exception(String msg, Throwable cause)`.

### Q14: Что такое StackOverflowError и когда он возникает?

- **ID:** `exc-medium-07`
- **Difficulty:** medium
- **Tags:** StackOverflowError, рекурсия, стек

**Варианты:**
- A) Checked исключение при нехватке heap-памяти
- B) Error (не Exception!), возникающий при переполнении стека вызовов — обычно при бесконечной рекурсии
- C) Исключение при обращении к null-ссылке
- D) RuntimeException при делении на ноль

**Ответ:** `B`

**Объяснение:** StackOverflowError — это Error (не Exception!), наследник VirtualMachineError. Каждый вызов метода добавляет фрейм в стек; при бесконечной рекурсии стек переполняется. Технически можно поймать в `catch(Error e)`, но это редко правильное решение — нужно исправить код.

### Q15: Что произойдёт с исключением из блока try, если в блоке finally тоже выбрасывается исключение?

- **ID:** `exc-hard-01`
- **Difficulty:** hard
- **Tags:** finally, подавление-исключений

**Варианты:**
- A) Оба исключения передаются вызывающему коду одновременно
- B) Исключение из finally подавляет (заменяет) исключение из try — первое теряется
- C) Исключение из try всегда имеет приоритет и передаётся дальше
- D) JVM завершает работу при конфликте двух исключений

**Ответ:** `B`

**Объяснение:** Если в `try` и в `finally` оба выбрасывают исключения — исключение из `finally` подавляет исключение из `try`. Исключение из `try` теряется. Это классическая ловушка. В try-with-resources (Java 7+) ситуация обратная: исключение из `try` — главное, исключения из `close()` добавляются как suppressed через `addSuppressed()`.

### Q16: Что такое suppressed exceptions и как получить к ним доступ?

- **ID:** `exc-hard-02`
- **Difficulty:** hard
- **Tags:** suppressed-exceptions, try-with-resources

**Варианты:**
- A) Исключения, намеренно проигнорированные разработчиком через пустой catch
- B) Исключения, добавленные к главному через addSuppressed(); возникают в try-with-resources при исключении в close()
- C) Checked исключения без объявления в throws — компилятор их подавляет
- D) Исключения из фоновых потоков, не связанные с основным потоком

**Ответ:** `B`

**Объяснение:** В try-with-resources: если в `try` исключение AND в `close()` исключение — главным является исключение из `try`, а исключение из `close()` добавляется как suppressed: `mainException.addSuppressed(closeException)`. Доступ: `e.getSuppressed()` возвращает массив подавленных исключений.

### Q17: Почему плохая практика ловить Exception или Throwable в catch?

- **ID:** `exc-hard-03`
- **Difficulty:** hard
- **Tags:** best-practices, catch-Exception, Throwable

**Варианты:**
- A) Это вызовет ошибку компиляции в современных версиях Java
- B) Это перехватит и скроет ошибки JVM (OutOfMemoryError, StackOverflowError), прервёт корректное завершение потоков и замаскирует реальные проблемы
- C) Производительность снижается при перехвате широких типов
- D) Нет никаких проблем — это рекомендуемая практика для надёжного кода

**Ответ:** `B`

**Объяснение:** `catch(Exception e)`: перехватывает NPE, ClassCastException — ошибки программиста, которые нужно исправлять, а не скрывать. `catch(Throwable t)`: перехватывает ThreadDeath, OutOfMemoryError — серьёзные сбои JVM, от которых нельзя корректно восстановиться. Лучше: ловить конкретные типы исключений, которые умеешь обрабатывать.

### Q18: Как правильно логировать исключения чтобы не терять информацию?

- **ID:** `exc-hard-04`
- **Difficulty:** hard
- **Tags:** логирование, best-practices, cause

**Варианты:**
- A) `catch(Exception e) { log.error(e.getMessage()); throw e; }`
- B) `catch(Exception e) { log.error("Error occurred"); }` — без стека
- C) `catch(Exception e) { log.error("Operation failed", e); }` — передать весь объект исключения в логгер
- D) Логировать и повторно бросать одновременно — двойное логирование лучше

**Ответ:** `C`

**Объяснение:** `log.error("message", e)` — передаёт весь объект исключения. Большинство логгеров (SLF4J, Log4j) выведут полный стек вызовов. `e.getMessage()` — теряет стек. Двойное логирование (поймал-залогировал-бросил-поймал-залогировал снова) засоряет логи. Логируй один раз на нужном уровне. Не проглатывай исключения в пустом catch без логирования.

### Q19: Что такое checked exceptions и почему их критикуют?

- **ID:** `exc-hard-05`
- **Difficulty:** hard
- **Tags:** checked, design, controversy

**Варианты:**
- A) Их критикуют за слишком высокую производительность
- B) Checked exceptions — уникальная особенность Java; критика: вынуждают загрязнять сигнатуры методов throws, нарушают принцип закрытости, часто проглатываются в пустом catch
- C) Они полностью устарели и удалены в Java 17+
- D) Их критикуют только потому что они сложны для понимания новичков

**Ответ:** `B`

**Объяснение:** Checked exceptions — уникальная фича Java. Аргументы ЗА: компилятор гарантирует обработку ошибок. Аргументы ПРОТИВ: 1) Загрязняют сигнатуры — `throws IOException` распространяется по всей цепочке. 2) Часто проглатываются: `catch(Exception e) {}`. 3) Нарушают инкапсуляцию деталей реализации. 4) Kotlin, C#, Python — не имеют checked и успешны. Современный Java-код часто использует unchecked.

### Q20: Как реализовать безопасный retry-механизм с использованием исключений?

- **ID:** `exc-hard-06`
- **Difficulty:** hard
- **Tags:** retry, паттерны, best-practices

**Варианты:**
- A) Ловить все исключения в бесконечном цикле пока не успешно
- B) Ловить только конкретные recoverable-исключения; ограничить число попыток; добавить задержку между попытками; после исчерпания — выбросить итоговое исключение с причиной
- C) Использовать finally для повторного вызова метода при ошибке
- D) Retry-механизм реализуется только через внешние библиотеки, вручную нельзя

**Ответ:** `B`

**Объяснение:** Корректный retry: `int attempts = 0; Exception lastEx = null; while(attempts < MAX_RETRIES) { try { return doOperation(); } catch(TransientException e) { lastEx = e; attempts++; Thread.sleep(DELAY * attempts); } } throw new RetryExhaustedException("Failed after " + MAX_RETRIES + " attempts", lastEx);`. Ключевые принципы: ловить только transient (временные) ошибки; передавать последнее исключение как cause; не retry при business-логических ошибках.
