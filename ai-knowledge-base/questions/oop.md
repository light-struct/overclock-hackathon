## 🟢 Лёгкие (`easy`)

---

### Q1: Что такое инкапсуляция?

- **ID:** `oop-easy-01`
- **Difficulty:** easy
- **Tags:** инкапсуляция, основы

**Варианты:**
- A) Возможность объекта принимать разные формы
- B) Скрытие внутренней реализации и предоставление публичного интерфейса
- C) Наследование свойств от родительского класса
- D) Разбиение программы на процедуры

**Ответ:** `B`

**Объяснение:** Инкапсуляция — это принцип, при котором внутреннее состояние объекта скрыто от внешнего кода. Доступ к данным предоставляется только через публичный интерфейс (методы). Это защищает данные от некорректного использования.

---

### Q2: Что такое объект в ООП?

- **ID:** `oop-easy-02`
- **Difficulty:** easy
- **Tags:** объект, основы

**Варианты:**
- A) Описание структуры данных без значений
- B) Функция с возвращаемым значением
- C) Экземпляр класса, созданный в памяти
- D) Абстрактный тип данных

**Ответ:** `C`

**Объяснение:** Объект — это конкретный экземпляр класса. Класс — это шаблон (чертёж), а объект — конкретная сущность созданная по этому шаблону, живущая в памяти и имеющая своё состояние.

---

### Q3: Что такое класс в ООП?

- **ID:** `oop-easy-03`
- **Difficulty:** easy
- **Tags:** класс, основы

**Варианты:**
- A) Конкретный экземпляр объекта в памяти
- B) Переменная, хранящая ссылку на объект
- C) Набор глобальных функций
- D) Шаблон/чертёж для создания объектов

**Ответ:** `D`

**Объяснение:** Класс — это шаблон, описывающий состояние (поля) и поведение (методы) объектов. По одному классу можно создать множество объектов — экземпляров этого класса.

---

### Q4: Что такое наследование в ООП?

- **ID:** `oop-easy-04`
- **Difficulty:** easy
- **Tags:** наследование, основы

**Варианты:**
- A) Создание нескольких экземпляров одного класса
- B) Скрытие деталей реализации класса
- C) Механизм, при котором класс получает свойства и методы другого класса
- D) Реализация нескольких интерфейсов одновременно

**Ответ:** `C`

**Объяснение:** Наследование позволяет одному классу (подклассу) получить поля и методы другого класса (суперкласса). Это способствует повторному использованию кода и построению иерархий.

---

### Q5: Что такое полиморфизм?

- **ID:** `oop-easy-05`
- **Difficulty:** easy
- **Tags:** полиморфизм, основы

**Варианты:**
- A) Сокрытие данных внутри класса
- B) Создание копии объекта
- C) Статическая типизация данных
- D) Способность объектов разных классов обрабатываться через единый интерфейс

**Ответ:** `D`

**Объяснение:** Полиморфизм позволяет использовать объекты разных классов одинаковым образом через общий интерфейс или суперкласс. Один и тот же метод может вести себя по-разному в зависимости от типа объекта.

---

### Q6: Что такое абстракция в ООП?

- **ID:** `oop-easy-06`
- **Difficulty:** easy
- **Tags:** абстракция, основы

**Варианты:**
- A) Объект может содержать другие объекты
- B) Выделение существенных характеристик объекта, игнорируя несущественные детали
- C) Каждый класс имеет только один экземпляр
- D) Методы не могут изменять состояние объекта

**Ответ:** `B`

**Объяснение:** Абстракция — это выделение только тех характеристик объекта, которые важны в данном контексте, и скрытие деталей реализации. Например, при работе с автомобилем нас интересует интерфейс (руль, педали), а не устройство двигателя.

---

### Q7: Что такое интерфейс в контексте ООП?

- **ID:** `oop-easy-07`
- **Difficulty:** easy
- **Tags:** интерфейс, абстракция

**Варианты:**
- A) Реализация класса с конкретными методами
- B) Переменная экземпляра класса
- C) Контракт, определяющий набор методов без реализации
- D) Метод, доступный только внутри класса

**Ответ:** `C`

**Объяснение:** Интерфейс — это контракт, который говорит "что умеет делать объект", не описывая "как именно". Классы, реализующие интерфейс, обязаны реализовать все его методы.

## 🟡 Средние (`medium`)

---

### Q8: Чем отличается перегрузка (overloading) от переопределения (overriding)?

- **ID:** `oop-medium-01`
- **Difficulty:** medium
- **Tags:** полиморфизм, overloading, overriding

**Варианты:**
- A) Это одно и то же понятие, просто разные термины
- B) Перегрузка работает только с конструкторами
- C) Переопределение — одинаковое имя, разные параметры в одном классе
- D) Перегрузка — одинаковое имя, разные параметры в одном классе; Переопределение — изменение метода в подклассе

**Ответ:** `D`

**Объяснение:** Overloading (перегрузка) — несколько методов с одним именем, но разными параметрами в одном классе. Overriding (переопределение) — подкласс предоставляет свою реализацию метода, объявленного в суперклассе, с той же сигнатурой.

### Q9: Что такое принцип Single Responsibility из SOLID?

- **ID:** `oop-medium-02`
- **Difficulty:** medium
- **Tags:** SOLID, SRP

**Варианты:**
- A) Класс должен быть открыт для расширения, закрыт для модификации
- B) Класс должен иметь одну и только одну причину для изменения
- C) Подкласс должен заменять суперкласс без нарушения поведения
- D) Зависимости должны строиться на абстракциях

**Ответ:** `B`

**Объяснение:** Single Responsibility Principle (SRP) — у класса должна быть одна ответственность, одна причина для изменения. Нарушение SRP: класс UserService одновременно работает с БД, отправляет email и форматирует отчёты.

---

### Q10: Что такое принцип Open/Closed из SOLID?

- **ID:** `oop-medium-03`
- **Difficulty:** medium
- **Tags:** SOLID, OCP

**Варианты:**
- A) Код должен быть открыт для чтения, закрыт для изменения
- B) Класс открыт для расширения, но закрыт для модификации существующего кода
- C) Методы открыты внутри пакета, закрыты снаружи
- D) Все классы должны быть публичными

**Ответ:** `B`

**Объяснение:** Open/Closed Principle: код должен быть открыт для расширения (добавлять новое поведение) и закрыт для модификации (не менять существующий работающий код). Достигается через абстракции и полиморфизм.

### Q11: Что такое агрегация в ООП?

- **ID:** `oop-medium-04`
- **Difficulty:** medium
- **Tags:** агрегация, ассоциация, связи-между-классами

**Варианты:**
- A) Один класс наследует другой
- B) Объект содержит ссылки на другие независимые объекты (слабое владение)
- C) Метод переопределяет родительский метод
- D) Класс реализует интерфейс

**Ответ:** `B`

**Объяснение:** Агрегация — тип ассоциации, где объект содержит другие объекты, но они независимы: могут существовать без него. Например, Университет содержит Студентов, но студенты существуют и после закрытия университета.

### Q12: Что такое композиция в ООП?

- **ID:** `oop-medium-05`
- **Difficulty:** medium
- **Tags:** композиция, связи-между-классами

**Варианты:**
- A) Класс имеет несколько конструкторов
- B) Методы класса вызывают методы того же класса
- C) Объект полностью владеет другими объектами — дочерние не живут без родителя
- D) Интерфейс наследует другой интерфейс

**Ответ:** `C`

**Объяснение:** Композиция — сильная форма агрегации. Дочерние объекты принадлежат родителю и уничтожаются вместе с ним. Например, Дом содержит Комнаты — при сносе дома комнат больше нет.

---

### Q13: Что означает принцип подстановки Лисков (LSP)?

- **ID:** `oop-medium-06`
- **Difficulty:** medium
- **Tags:** SOLID, LSP, наследование

**Варианты:**
- A) Методы суперкласса должны быть закрытыми (private)
- B) Подкласс должен быть полностью заменяем своим суперклассом без нарушения поведения программы
- C) Интерфейсы должны быть минимальными
- D) Зависимости должны строиться на абстракциях

**Ответ:** `B`

**Объяснение:** LSP (Liskov Substitution Principle): если S — подтип T, то объекты типа T могут быть заменены объектами типа S без изменения поведения программы. Нарушение LSP: Square extends Rectangle, но изменение стороны квадрата нарушает поведение Rectangle.

### Q14: Что такое связность (cohesion) класса?

- **ID:** `oop-medium-07`
- **Difficulty:** medium
- **Tags:** cohesion, coupling, качество-кода

**Варианты:**
- A) Число зависимостей класса от других классов
- B) Количество полей класса
- C) Глубина иерархии наследования
- D) Степень, в которой методы класса связаны с его единственной целью

**Ответ:** `D`

**Объяснение:** Высокая cohesion (сплочённость) — все методы класса работают для одной цели. Это хорошо. Высокий coupling (связность с другими) — плохо. Идеал: высокий cohesion + низкий coupling.

### Q15: Что такое Dependency Inversion Principle (DIP)?

- **ID:** `oop-hard-01`
- **Difficulty:** hard
- **Tags:** SOLID, DIP, абстракция

**Варианты:**
- A) Каждый класс должен зависеть от одного интерфейса
- B) Метод не должен принимать более трёх параметров
- C) Зависимости должны строиться на абстракциях, а не на конкретных реализациях
- D) Все зависимости должны быть приватными полями

**Ответ:** `C`

**Объяснение:** DIP: модули верхнего уровня не должны зависеть от модулей нижнего уровня — оба должны зависеть от абстракций. Абстракции не должны зависеть от деталей. Это основа для Dependency Injection (DI).

### Q16: Что такое «God Object» — антипаттерн в ООП?

- **ID:** `oop-hard-02`
- **Difficulty:** hard
- **Tags:** антипаттерны, качество-кода, SRP

**Варианты:**
- A) Паттерн Singleton с глобальным состоянием
- B) Объект/класс, который знает слишком много и делает слишком много
- C) Абстрактный базовый класс для всей системы
- D) Класс с только статическими методами и состоянием

**Ответ:** `B`

**Объяснение:** God Object — антипаттерн, когда один класс берёт на себя слишком много ответственностей. Нарушает SRP, делает код трудным для тестирования и сопровождения. Решение: разбить на несколько классов с чёткими зонами ответственности.

### Q17: Какой принцип ООП нарушается, если метод getInternalList() возвращает прямую ссылку на внутренний список?

- **ID:** `oop-hard-03`
- **Difficulty:** hard
- **Tags:** инкапсуляция, defensive-copy

**Варианты:**
- A) Полиморфизм
- B) Наследование
- C) Инкапсуляция
- D) Абстракция

**Ответ:** `C`

**Объяснение:** Возврат прямой ссылки на внутреннее поле нарушает инкапсуляцию — внешний код может модифицировать внутреннее состояние объекта в обход его методов. Решение: возвращать defensive copy (Collections.unmodifiableList() или new ArrayList<>(internalList)).

### Q18: В чём разница между Interface Segregation Principle (ISP) и принципом единственной ответственности (SRP)?

- **ID:** `oop-hard-04`
- **Difficulty:** hard
- **Tags:** SOLID, ISP, SRP

**Варианты:**
- A) Это одно и то же, просто применяется к разным конструкциям
- B) SRP — про классы (одна ответственность), ISP — про интерфейсы (не заставлять реализовывать ненужные методы)
- C) ISP применяется только к абстрактным классам
- D) SRP запрещает наследование, ISP — множественную реализацию

**Ответ:** `B`

**Объяснение:** SRP говорит что класс должен иметь одну причину для изменения. ISP говорит что клиент не должен зависеть от методов интерфейса, которые он не использует. Лучше много маленьких специализированных интерфейсов, чем один толстый (fat interface).

### Q19: Что такое «утиная типизация» (Duck Typing) и как это связано с полиморфизмом?

- **ID:** `oop-hard-05`
- **Difficulty:** hard
- **Tags:** полиморфизм, duck-typing, типизация

**Варианты:**
- A) Тип объекта определяется только именем класса
- B) Объект подходит под тип, если имеет нужные методы/свойства — независимо от его класса в иерархии
- C) Только примитивные типы используются в строго типизированных системах
- D) Все объекты приводятся к общему базовому типу Object

**Ответ:** `B`

**Объяснение:** Duck Typing: "если оно ходит как утка и крякает как утка — это утка". Объект совместим с типом, если имеет нужный интерфейс (методы), независимо от иерархии наследования. Это форма структурного полиморфизма, характерная для Python/JS, но реализуемая через интерфейсы в Java.

### Q20: Как правильно применить паттерн «Программируй к интерфейсу, а не к реализации»?

- **ID:** `oop-hard-06`
- **Difficulty:** hard
- **Tags:** интерфейс, паттерны, dependency-injection

**Варианты:**
- A) Использовать только стандартные Java-интерфейсы из JDK
- B) Все поля класса должны быть объявлены как Object
- C) Объявлять переменные и параметры типом интерфейса, а не конкретного класса; передавать реализацию извне
- D) Никогда не использовать абстрактные классы, только интерфейсы

**Ответ:** `C`

**Объяснение:** Вместо `ArrayList<String> list = new ArrayList<>()` пиши `List<String> list = new ArrayList<>()`. Вместо зависимости от MySQLRepository внутри сервиса — принимай Repository (интерфейс) через конструктор. Это позволяет легко подменить реализацию (например, в тестах) без изменения зависимого кода.
