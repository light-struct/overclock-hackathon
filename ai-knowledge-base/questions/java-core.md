### Q1: Что такое JVM?

- **ID:** `java-easy-01`
- **Difficulty:** easy
- **Tags:** JVM, архитектура-java

**Варианты:**
- A) Java Version Manager — менеджер версий Java
- B) Java Verification Module — модуль проверки кода
- C) Java Virtual Machine — виртуальная машина, исполняющая байткод Java
- D) Java Visual Manager — инструмент для построения GUI

**Ответ:** `C`

**Объяснение:** JVM (Java Virtual Machine) — виртуальная машина, которая исполняет скомпилированный байткод Java (.class файлы). Именно JVM обеспечивает принцип "Write Once, Run Anywhere" — один и тот же байткод работает на любой платформе, где установлена JVM.

### Q2: Чем отличается JDK от JRE?

- **ID:** `java-easy-02`
- **Difficulty:** easy
- **Tags:** JDK, JRE, JVM, архитектура-java

**Варианты:**
- A) JDK — для запуска программ, JRE — для разработки
- B) JDK включает JRE + компилятор (javac) и инструменты разработки; JRE — только среда выполнения
- C) Это одно и то же, просто разные названия
- D) JRE содержит JDK как вложенный компонент

**Ответ:** `B`

**Объяснение:** JRE (Java Runtime Environment) = JVM + стандартные библиотеки — только для запуска программ. JDK (Java Development Kit) = JRE + компилятор javac + отладчик + другие инструменты — для разработки. С Java 11 JRE отдельно не поставляется.

### Q3: Почему String является неизменяемым (immutable) в Java?

- **ID:** `java-easy-03`
- **Difficulty:** easy
- **Tags:** String, immutable, String-Pool

**Варианты:**
- A) Это техническое ограничение JVM
- B) Для безопасности, кэширования hashCode и возможности использования в String Pool
- C) Строки хранятся в стеке, где нельзя изменять данные
- D) Из-за отсутствия сеттеров в классе String

**Ответ:** `B`

**Объяснение:** String immutable по трём причинам: 1) Безопасность — нельзя изменить строку после передачи (пароли, URL). 2) String Pool — одинаковые строки-литералы разделяют один объект в памяти. 3) HashCode кэшируется при первом вызове и всегда одинаков.

### Q4: В чём разница между == и .equals() для объектов?

- **ID:** `java-easy-04`
- **Difficulty:** easy
- **Tags:** equals, hashCode, сравнение

**Варианты:**
- A) Нет никакой разницы для объектов
- B) == сравнивает ссылки (адреса в памяти), equals() — содержимое (если переопределён)
- C) equals() сравнивает ссылки, == — содержимое объектов
- D) == работает только для примитивов, объекты сравнивать нельзя

**Ответ:** `B`

**Объяснение:** `==` проверяет, указывают ли две переменные на один и тот же объект в памяти. `equals()` по умолчанию делает то же самое (Object.equals), но обычно переопределяется для сравнения по содержимому. Например, `"hello" == "hello"` может быть true (String Pool), но `new String("hello") == new String("hello")` — false.

### Q5: Что такое autoboxing в Java?

- **ID:** `java-easy-05`
- **Difficulty:** easy
- **Tags:** autoboxing, примитивы, обёртки

**Варианты:**
- A) Автоматическое создание новых потоков выполнения
- B) Автоматическое преобразование примитивного типа в объект-обёртку и обратно
- C) Автоматическая инициализация переменных значениями по умолчанию
- D) Автоматическое выравнивание объектов в памяти

**Ответ:** `B`

**Объяснение:** Autoboxing: `int → Integer` автоматически. Unboxing: `Integer → int` автоматически. Пример: `List<Integer> list = new ArrayList<>(); list.add(5);` — здесь 5 автоматически упаковывается в `Integer.valueOf(5)`. Может вызвать NullPointerException при unboxing null.

### Q6: Что такое generics в Java?

- **ID:** `java-easy-06`
- **Difficulty:** easy
- **Tags:** generics, типизация

**Варианты:**
- A) Набор встроенных утилитных методов класса Collections
- B) Автоматическое приведение типов во время выполнения
- C) Механизм создания универсального кода с параметризованными типами
- D) Интерфейс для работы с коллекциями

**Ответ:** `C`

**Объяснение:** Generics позволяют создавать классы, методы и интерфейсы с параметрами типов: `List<String>`, `Map<K,V>`. Это обеспечивает типобезопасность во время компиляции и устраняет необходимость явного приведения типов. Type erasure: информация о generic-типе стирается во время компиляции.

### Q7: Что такое static поле класса?

- **ID:** `java-easy-07`
- **Difficulty:** easy
- **Tags:** static, поля-класса

**Варианты:**
- A) Поле, которое нельзя изменить после инициализации
- B) Поле принадлежит классу, а не экземпляру — общее для всех объектов класса
- C) Поле, которое не участвует в сериализации
- D) Поле, доступное только внутри самого класса

**Ответ:** `B`

**Объяснение:** `static` поле существует в единственном экземпляре для всего класса. Все объекты класса разделяют одно и то же static поле. Хранится в области Method Area (Metaspace), а не в heap вместе с объектом. Вызывается через имя класса: `ClassName.field`.

### Q8: Как работает HashMap внутри?

- **ID:** `java-medium-01`
- **Difficulty:** medium
- **Tags:** HashMap, collections, hashCode, equals

**Варианты:**
- A) Как отсортированный список пар ключ-значение
- B) Использует массив бакетов; коллизии разрешаются связными списками (Java 7) или деревьями (Java 8+) на основе hashCode и equals
- C) Хранит данные исключительно в бинарном дереве поиска
- D) Хранит данные в очереди FIFO по порядку добавления

**Ответ:** `B`

**Объяснение:** HashMap: массив бакетов (bucket array). Ключ хешируется: `index = hashCode(key) % capacity`. Коллизии (одинаковый индекс) — в Java 7 связный список, в Java 8+ при > 8 элементов в бакете список превращается в красно-чёрное дерево (TreeBin). Операции O(1) в среднем, O(n) при деградации.

### Q9: Что такое контракт equals/hashCode?

- **ID:** `java-medium-02`
- **Difficulty:** medium
- **Tags:** equals, hashCode, контракт

**Варианты:**
- A) Если hashCode одинаковый — equals обязан вернуть true
- B) Если equals() вернул true — hashCode() обязан вернуть одинаковое значение для обоих объектов
- C) equals и hashCode никак не связаны между собой
- D) hashCode всегда должен возвращать 0 для корректной работы HashMap

**Ответ:** `B`

**Объяснение:** Контракт: если a.equals(b) == true, то a.hashCode() == b.hashCode(). Обратное необязательно (коллизии допустимы). Нарушение приводит к потере объектов в HashMap/HashSet — put добавит, но get не найдёт, потому что поиск идёт сначала по hashCode.

### Q10: В чём разница между ArrayList и LinkedList?

- **ID:** `java-medium-03`
- **Difficulty:** medium
- **Tags:** ArrayList, LinkedList, collections, сложность-алгоритмов

**Варианты:**
- A) Нет никакой разницы, это синонимы
- B) ArrayList — на основе массива (быстрый get по индексу O(1)); LinkedList — двусвязный список (быстрая вставка/удаление в середине O(1), но медленный get O(n))
- C) LinkedList быстрее при любых операциях
- D) ArrayList не поддерживает дублирующие элементы

**Ответ:** `B`

**Объяснение:** ArrayList: O(1) — get(i), O(n) — insert/remove в середине (сдвиг элементов). LinkedList: O(1) — add/remove в начале/конце, O(n) — get(i) (обход от начала). На практике ArrayList быстрее в большинстве случаев из-за locality of reference (кэш CPU).

### Q11: Что такое Stream API в Java 8?

- **ID:** `java-medium-04`
- **Difficulty:** medium
- **Tags:** stream-api, java8, функциональное-программирование

**Варианты:**

- B) API для функциональной обработки последовательностей данных с методами map/filter/reduce
- C) Многопоточная обработка только через параллельные потоки
- D) Устаревшая замена интерфейса Iterator

**Ответ:** `B`

### Q12: Что такое Optional в Java 8?

- **ID:** `java-medium-05`
- **Difficulty:** medium
- **Tags:** Optional, null, java8

**Варианты:**
- A) Необязательный параметр метода со значением по умолчанию
- B) Контейнер, который может содержать или не содержать значение — явная замена null
- C) Аннотация для пометки nullable-параметров
- D) Ленивая инициализация поля при первом обращении

**Ответ:** `B`

**Объяснение:** `Optional<T>` — обёртка, явно выражающая что значение может отсутствовать. Методы: `Optional.of(val)`, `Optional.empty()`, `Optional.ofNullable(val)`. Использование: `.isPresent()`, `.get()`, `.orElse(default)`, `.ifPresent(consumer)`. Цель — избежать NullPointerException через явную обработку отсутствия значения.

### Q13: Что такое принцип fail-fast в Java коллекциях?

- **ID:** `java-medium-06`
- **Difficulty:** medium
- **Tags:** fail-fast, collections, ConcurrentModificationException

**Варианты:**
- A) Коллекция автоматически исправляет ошибки при итерации
- B) Итератор бросает ConcurrentModificationException если коллекция изменилась во время итерации
- C) Метод возвращает null вместо исключения при ошибке
- D) JVM немедленно завершает работу при первой ошибке в коллекции

**Ответ:** `B`

**Объяснение:** Fail-fast итераторы (ArrayList, HashMap и др.) отслеживают поле `modCount`. При итерации проверяют: если modCount изменился (добавление/удаление элементов) — бросают `ConcurrentModificationException`. Альтернатива: CopyOnWriteArrayList (fail-safe, итерируется по копии).

### Q14: Что возвращает метод hashCode() по умолчанию в Object?

- **ID:** `java-medium-07`
- **Difficulty:** medium
- **Tags:** hashCode, Object, JVM

**Варианты:**
- A) Уникальный инкрементальный ID объекта начиная с 1
- B) Целочисленный хэш, основанный на адресе объекта в памяти (или внутреннем идентификаторе JVM)

### Q15: Что такое String Pool и как работает intern()?

- **ID:** `java-hard-01`
- **Difficulty:** hard
- **Tags:** String-Pool, intern, память

**Варианты:**
- A) String Pool — это очередь для многопоточной обработки строк
- B) String Pool — область в heap (с Java 7+ в heap, ранее в PermGen) для уникальных строк-литералов; intern() помещает строку в пул и возвращает ссылку на неё
- C) String Pool хранится в стеке и очищается после выхода из метода
- D) intern() создаёт копию строки в нативной памяти

**Ответ:** `B`

**Объяснение:** Строковые литералы `"hello"` автоматически помещаются в String Pool. `new String("hello")` создаёт новый объект вне пула. `str.intern()` — если строка уже в пуле, вернёт ссылку на неё; иначе добавит и вернёт. С Java 7 пул переехал из PermGen в heap (GC может очищать).

### Q16: Как работает type erasure в generics?

- **ID:** `java-hard-02`
- **Difficulty:** hard
- **Tags:** generics, type-erasure, JVM

**Варианты:**
- A) Generic-информация сохраняется в байткоде и доступна в runtime
- B) Компилятор стирает generic-параметры и заменяет их на Object (или upper bound); проверки только на этапе компиляции
- C) Type erasure происходит только для примитивных типов
- D) JVM создаёт отдельный класс для каждой generic-специализации (как в C++)

**Ответ:** `B`

**Объяснение:** Type Erasure: `List<String>` и `List<Integer>` в байткоде — одинаковый `List`. Компилятор добавляет проверки и касты автоматически. Следствие: нельзя `new T()`, `new T[]`, `instanceof List<String>`, нельзя перегружать методы по generic-типу.

### Q17: В чём разница между Comparable и Comparator?

- **ID:** `java-hard-03`
- **Difficulty:** hard
- **Tags:** Comparable, Comparator, сортировка

**Варианты:**
- A) Это одно и то же, просто разные интерфейсы для одной цели
- B) Comparable — естественный порядок объекта (метод compareTo в самом классе); Comparator — внешняя стратегия сортировки (отдельный объект/лямбда)
- C) Comparator используется только для примитивов
- D) Comparable работает только с Collections.sort(), Comparator — только с Arrays.sort()

**Ответ:** `B`

**Объяснение:** `Comparable<T>`: класс реализует `compareTo(T o)` — определяет "естественный" порядок (String, Integer это делают). `Comparator<T>`: внешний объект с методом `compare(T o1, T o2)` — определяет альтернативный порядок. `Collections.sort(list)` требует Comparable; `Collections.sort(list, comparator)` — Comparator.

### Q18: Что такое var и каковы его ограничения (Java 10+)?

- **ID:** `java-hard-04`
- **Difficulty:** hard
- **Tags:** var, local-variable-type-inference, java10

**Варианты:**
- A) var позволяет создавать переменные без типа, как в JavaScript
- B) var — вывод типа локальной переменной компилятором; тип определяется из правой части, переменная всё равно строго типизирована
- C) var можно использовать для полей класса и параметров методов
- D) var автоматически делает переменную nullable

**Ответ:** `B`

**Объяснение:** `var x = new ArrayList<String>()` — компилятор выводит тип `ArrayList<String>`. Ограничения: только локальные переменные с инициализацией; нельзя для полей класса, параметров методов, типов возврата; нельзя `var x = null`. Java остаётся строго типизированной — тип фиксируется на этапе компиляции.

### Q19: Как работает ConcurrentHashMap в отличие от HashMap?

- **ID:** `java-hard-05`
- **Difficulty:** hard
- **Tags:** ConcurrentHashMap, многопоточность, collections

**Варианты:**
- A) ConcurrentHashMap синхронизирует весь объект целиком как Collections.synchronizedMap()
- B) ConcurrentHashMap использует сегментную блокировку (Java 7) или CAS + синхронизацию на уровне бакета (Java 8+) — высокая конкурентность без блокировки всей карты
- C) ConcurrentHashMap не допускает null-ключей и null-значений из соображений безопасности
- D) Ответы B и C верны

**Ответ:** `D`

**Объяснение:** ConcurrentHashMap (Java 8+): использует CAS (Compare-And-Swap) для атомарных операций и синхронизированные блоки только при необходимости на уровне отдельных бакетов. Не допускает null-ключей и null-значений (в отличие от HashMap) — null создаёт неоднозначность в многопоточной среде.

### Q20: Что такое lambda capturing и какие переменные можно захватить?

- **ID:** `java-hard-06`
- **Difficulty:** hard
- **Tags:** lambda, closure, effectively-final

**Варианты:**
- A) Lambda может захватывать любые локальные переменные без ограничений
- B) Lambda может захватывать только final или effectively final локальные переменные; поля класса и static переменные — без ограничений
- C) Lambda не может захватывать никаких переменных из внешнего контекста
- D) Lambda может изменять захваченные локальные переменные через массив-обёртку

**Ответ:** `B`

**Объяснение:** Lambda захватывает только `final` или effectively final (не изменяемые после инициализации) локальные переменные. Причина: локальные переменные хранятся в стеке потока — lambda может жить дольше метода, поэтому JVM копирует значение. Изменение нарушило бы консистентность. Поля объекта (`this.field`) и `static` поля захватываются свободно, так как хранятся в heap.
