# Сериализация и Копирование — Банк вопросов

> **Тема:** Сериализация и копирование объектов в Java  
> **Префикс ID:** `serial`  
> **Всего вопросов:** 20 (7 лёгких · 7 средних · 6 сложных)

## 🟢 Лёгкие (`easy`)

---

### Q1: Что такое сериализация в Java?

- **ID:** `serial-easy-01`
- **Difficulty:** easy
- **Tags:** сериализация, основы

**Варианты:**
- A) Шифрование объекта для безопасной передачи по сети
- B) Процесс преобразования объекта в поток байт для сохранения или передачи
- C) Сжатие объекта в памяти для экономии места
- D) Преобразование объекта в JSON-строку

**Ответ:** `B`

**Объяснение:** Сериализация — преобразование объекта Java в поток байт, который можно записать в файл, передать по сети или сохранить в БД. Десериализация — обратный процесс: восстановление объекта из потока байт. Оба механизма встроены в Java через интерфейс `Serializable`.

---

### Q2: Какой интерфейс нужно реализовать для сериализации объекта?

- **ID:** `serial-easy-02`
- **Difficulty:** easy
- **Tags:** Serializable, интерфейсы

**Варианты:**
- A) Только Serializable
- B) Только Cloneable
- C) Serializable или Externalizable
- D) Только Externalizable

**Ответ:** `C`

**Объяснение:** Для сериализации класс должен реализовать `Serializable` (маркерный интерфейс, автоматическая сериализация) или `Externalizable` (полный контроль через `writeExternal`/`readExternal`). `Serializable` — проще и чаще используется.

---

### Q3: Что делает модификатор transient применительно к полю?

- **ID:** `serial-easy-03`
- **Difficulty:** easy
- **Tags:** transient, сериализация

**Варианты:**
- A) Делает поле только для чтения после инициализации
- B) Исключает поле из процесса сериализации
- C) Делает поле доступным для нескольких потоков одновременно
- D) Поле сериализуется в первую очередь перед остальными

**Ответ:** `B`

**Объяснение:** `transient` поле не включается в сериализованный поток байт. При десериализации такое поле получает значение по умолчанию (null, 0, false). Используется для: паролей, кэшируемых вычислений, полей без смысла вне контекста (Connection, Thread), полей несериализуемых типов.

---

### Q4: Что такое десериализация?

- **ID:** `serial-easy-04`
- **Difficulty:** easy
- **Tags:** десериализация, основы

**Варианты:**
- A) Удаление объекта из памяти после использования
- B) Восстановление объекта из потока байт
- C) Создание копии объекта в памяти
- D) Шифрование сериализованных данных

**Ответ:** `B`

**Объяснение:** Десериализация — обратный процесс к сериализации. Из потока байт восстанавливается объект Java с теми же полями и значениями. Важно: конструктор НЕ вызывается при десериализации — объект восстанавливается напрямую из байт.

---

### Q5: Какой класс используется для записи сериализованного объекта в файл?

- **ID:** `serial-easy-05`
- **Difficulty:** easy
- **Tags:** ObjectOutputStream, ввод-вывод

**Варианты:**
- A) FileInputStream
- B) DataOutputStream
- C) ObjectOutputStream
- D) BufferedWriter

**Ответ:** `C`

**Объяснение:** `ObjectOutputStream` — обёртка над любым OutputStream для записи объектов. Пример: `ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("data.ser")); oos.writeObject(myObject); oos.close();`. Для чтения обратно: `ObjectInputStream`.

---

### Q6: Что такое поверхностное копирование (shallow copy)?

- **ID:** `serial-easy-06`
- **Difficulty:** easy
- **Tags:** shallow-copy, копирование

**Варианты:**
- A) Создание нового объекта с полностью независимыми копиями всех вложенных объектов
- B) Создание нового объекта, поля которого указывают на те же объекты, что и оригинал
- C) Копирование только примитивных полей, остальные поля остаются null
- D) Клонирование объекта через механизм сериализации

**Ответ:** `B`

**Объяснение:** Shallow copy: новый объект создаётся, но его поля-ссылки указывают на те же объекты, что и у оригинала. Изменение вложенного объекта через копию изменит и оригинал (они разделяют ссылки). `Object.clone()` по умолчанию делает shallow copy.

---

### Q7: Какой интерфейс нужно реализовать чтобы использовать метод clone()?

- **ID:** `serial-easy-07`
- **Difficulty:** easy
- **Tags:** Cloneable, clone

**Варианты:**
- A) Serializable
- B) Copyable
- C) Cloneable
- D) Duplicable

**Ответ:** `C`

**Объяснение:** `Cloneable` — маркерный интерфейс без методов. Если класс не реализует `Cloneable`, вызов `clone()` бросит `CloneNotSupportedException`. Метод `clone()` объявлен в `Object` как `protected` — нужно переопределить как `public` в своём классе.

## 🟡 Средние (`medium`)

---

### Q8: Что такое serialVersionUID и зачем он нужен?

- **ID:** `serial-medium-01`
- **Difficulty:** medium
- **Tags:** serialVersionUID, версионирование

**Варианты:**
- A) Уникальный ID объекта, генерируемый при каждом создании экземпляра
- B) Статическое поле для проверки совместимости версии класса при десериализации
- C) Счётчик количества созданных экземпляров класса
- D) Хэш-код содержимого сериализованного объекта

**Ответ:** `B`

**Объяснение:** `private static final long serialVersionUID = 1L;` — если serialVersionUID в файле отличается от текущего класса, при десериализации бросается `InvalidClassException`. Если не объявить — JVM вычислит автоматически по структуре класса: любое изменение класса изменит UID и сломает десериализацию старых данных.

---

### Q9: Вызывается ли конструктор при десериализации объекта?

- **ID:** `serial-medium-02`
- **Difficulty:** medium
- **Tags:** десериализация, конструктор

**Варианты:**
- A) Да, всегда вызывается конструктор без аргументов
- B) Нет, объект восстанавливается напрямую из байт без вызова конструктора
- C) Да, вызывается любой доступный конструктор с null-аргументами
- D) Зависит от наличия конструктора без аргументов в классе

**Ответ:** `B`

**Объяснение:** При десериализации конструктор класса НЕ вызывается. JVM создаёт объект напрямую через `Unsafe.allocateInstance()` и заполняет поля из потока байт. Однако конструктор первого несериализуемого суперкласса в иерархии — вызывается. Это важно: проверки в конструкторе при десериализации не работают.

---

### Q10: Что такое глубокое копирование (deep copy)?

- **ID:** `serial-medium-03`
- **Difficulty:** medium
- **Tags:** deep-copy, копирование

**Варианты:**
- A) Копирование только ссылки на объект (самый быстрый способ)
- B) Полная рекурсивная копия объекта и всех вложенных объектов — полная независимость от оригинала
- C) Копирование только полей первого уровня (без рекурсии)
- D) Вызов метода clone() без переопределения

**Ответ:** `B`

**Объяснение:** Deep copy: новый объект создаётся, и все вложенные объекты тоже копируются рекурсивно. Изменение копии не влияет на оригинал. Реализация: 1) Явное копирование каждого поля. 2) Через сериализацию/десериализацию. 3) Через библиотеки (Apache Commons BeanUtils, Kryo).

---

### Q11: Как реализовать deep copy через сериализацию?

- **ID:** `serial-medium-04`
- **Difficulty:** medium
- **Tags:** deep-copy, сериализация, ByteArrayOutputStream

**Варианты:**
- A) Вызвать `Collections.copy()` для каждого поля
- B) Сериализовать объект в ByteArrayOutputStream, затем десериализовать из ByteArrayInputStream
- C) Вызвать `super.clone()` рекурсивно для всех вложенных объектов
- D) Использовать `Arrays.copyOf()` для каждого поля

**Ответ:** `B`

**Объяснение:** `ByteArrayOutputStream bos = new ByteArrayOutputStream(); new ObjectOutputStream(bos).writeObject(original); ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray()); T copy = (T) new ObjectInputStream(bis).readObject();` — полная deep copy. Требует чтобы все объекты в графе были Serializable. Медленнее явного копирования.

---

### Q12: Сериализуются ли статические (static) поля класса?

- **ID:** `serial-medium-05`
- **Difficulty:** medium
- **Tags:** static, сериализация

**Варианты:**
- A) Да, как и любые другие поля класса
- B) Нет, static поля принадлежат классу, а не объекту, и не включаются в поток сериализации
- C) Только если помечены специальной аннотацией
- D) Только примитивные static поля сериализуются

**Ответ:** `B`

**Объяснение:** Static поля принадлежат классу, а не конкретному объекту. Сериализация сохраняет состояние объекта — static поля в это не входят. Аналогично: `transient` поля тоже не сериализуются. При десериализации static поля будут иметь текущее значение из JVM, а не то что было при сериализации.

### Q13: Что выбрасывает Object.clone() если класс не реализует Cloneable?

- **ID:** `serial-medium-06`
- **Difficulty:** medium
- **Tags:** clone, Cloneable, исключения

**Варианты:**
- A) IllegalStateException
- B) UnsupportedOperationException
- C) CloneNotSupportedException
- D) SerializationException

**Ответ:** `C`

**Объяснение:** `Object.clone()` проверяет реализует ли объект `Cloneable`. Если нет — бросает `CloneNotSupportedException` (checked!). Именно поэтому при переопределении `clone()` нужно либо объявить `throws CloneNotSupportedException`, либо поймать и перебросить как unchecked.

### Q14: Чем Externalizable отличается от Serializable?

- **ID:** `serial-medium-07`
- **Difficulty:** medium
- **Tags:** Externalizable, Serializable, сравнение

**Варианты:**
- A) Externalizable — более безопасная версия Serializable
- B) Externalizable требует реализации writeExternal/readExternal — полный контроль над сериализацией; Serializable — автоматическая
- C) Externalizable используется только для XML-сериализации
- D) Externalizable — устаревший аналог Serializable из Java 1.0

**Ответ:** `B`

**Объяснение:** `Externalizable extends Serializable`. Требует реализации: `writeExternal(ObjectOutput)` и `readExternal(ObjectInput)`. При десериализации Externalizable ВЫЗЫВАЕТ конструктор без аргументов! Даёт полный контроль: что сериализовать и как. Более производительно, но больше кода. Serializable — автоматически, Externalizable — вручную.

### Q15: Что произойдёт с полями суперкласса при сериализации, если суперкласс не реализует Serializable?

- **ID:** `serial-hard-01`
- **Difficulty:** hard
- **Tags:** наследование, сериализация, суперкласс

**Варианты:**
- A) Поля суперкласса сериализуются автоматически вместе с остальными
- B) Поля суперкласса не сериализуются; при десериализации вызывается конструктор без аргументов несериализуемого суперкласса
- C) Выбрасывается NotSerializableException при попытке сериализации
- D) Поля суперкласса сохраняются как null

**Ответ:** `B`

**Объяснение:** Если класс A (не Serializable) → класс B (Serializable): при сериализации B, поля из A не сохраняются. При десериализации B: конструктор без аргументов A обязательно вызывается (A должен иметь public или protected конструктор без аргументов). Поля A инициализируются через этот конструктор.

### Q16: Как защитить чувствительные данные (пароли) при сериализации?

- **ID:** `serial-hard-02`
- **Difficulty:** hard
- **Tags:** безопасность, transient, writeObject

**Варианты:**
- A) Использовать шифрование всего объекта перед сериализацией
- B) Пометить чувствительные поля transient; при необходимости — реализовать writeObject/readObject для кастомной логики
- C) Использовать только Externalizable вместо Serializable
- D) Отключить сериализацию выбрасыванием исключения в writeObject

**Ответ:** `B`

**Объяснение:** Лучший подход: `private transient String password;` — поле не сериализуется. Дополнительно можно реализовать `private void writeObject(ObjectOutputStream oos)` и `readObject(ObjectInputStream ois)` для кастомной логики (шифрование, валидация). Для полного запрета сериализации: объявить `writeObject`/`readObject` которые бросают `NotSerializableException`.

### Q17: Что такое readResolve() и writeReplace() в сериализации?

- **ID:** `serial-hard-03`
- **Difficulty:** hard
- **Tags:** readResolve, writeReplace, Singleton, паттерны

**Варианты:**
- A) Методы для ручного управления потоком байт при сериализации
- B) writeReplace() — заменяет объект перед сериализацией; readResolve() — заменяет десериализованный объект после восстановления
- C) Методы из интерфейса Externalizable для кастомной логики
- D) Устаревшие методы, удалённые в Java 11

**Ответ:** `B`

**Объяснение:** `writeReplace()`: вызывается перед записью — можно вернуть другой объект для сериализации. `readResolve()`: вызывается после восстановления из потока — можно вернуть другой объект. Классический кейс: Singleton через сериализацию. Без `readResolve()` десериализация создаст второй экземпляр. С `readResolve()`: `return INSTANCE;` — всегда возвращает единственный экземпляр.

### Q18: Что такое serialization proxy pattern (паттерн сериализационного прокси)?

- **ID:** `serial-hard-04`
- **Difficulty:** hard
- **Tags:** serialization-proxy, паттерны, безопасность

**Варианты:**
- A) Использование прокси-сервера для передачи сериализованных объектов по сети
- B) Паттерн: сериализовать вспомогательный inner class (прокси) вместо оригинального объекта для безопасности и гибкости
- C) Автоматическая генерация serialVersionUID через прокси
- D) Кэширование сериализованных объектов для повторного использования

**Ответ:** `B`

**Объяснение:** Serialization Proxy: `writeReplace()` возвращает inner class SerializationProxy(originalObject). Proxy сериализуется вместо оригинала. Proxy.readResolve() восстанавливает оригинальный объект через публичный API. Преимущества: инварианты всегда проверяются через конструктор; защита от атак десериализации; работает с final-классами.

### Q19: Чем опасна десериализация ненадёжных данных?

- **ID:** `serial-hard-05`
- **Difficulty:** hard
- **Tags:** безопасность, десериализация, уязвимости

**Варианты:**
- A) Только снижает производительность из-за тяжёлой операции восстановления
- B) Злоумышленник может создать вредоносный поток байт, который выполнит произвольный код через цепочку гаджетов (deserialization gadget chain)
- C) Может вызвать несовместимость версий через неправильный serialVersionUID
- D) Десериализация всегда безопасна если поле помечено transient

**Ответ:** `B`

**Объяснение:** Десериализация ненадёжных данных — критическая уязвимость (OWASP Top 10). Атака: специально созданный поток байт заставляет JVM выполнить вредоносный код через цепочки вызовов в стандартных классах (gadget chains). Примеры: Apache Commons Collections RCE. Решения: не десериализовывать ненадёжные данные; использовать JSON/XML вместо Java serialization; фильтрация классов через `ObjectInputFilter` (Java 9+).

### Q20: Как реализовать Singleton, устойчивый к сериализации?

- **ID:** `serial-hard-06`
- **Difficulty:** hard
- **Tags:** Singleton, readResolve, паттерны

**Варианты:**
- A) Пометить поле INSTANCE как transient
- B) Реализовать readResolve() который возвращает INSTANCE; или использовать enum-Singleton
- C) Добавить synchronized в конструктор
- D) Сериализация Singleton невозможна — нужно отключить

**Ответ:** `B`

**Объяснение:** Без защиты: десериализация создаёт второй экземпляр Singleton. Решение 1: `private Object readResolve() { return INSTANCE; }` — после десериализации возвращает существующий экземпляр, новый объект отбрасывается GC. Решение 2 (лучшее): `enum Singleton { INSTANCE; }` — enum автоматически защищён от создания дополнительных экземпляров при сериализации/десериализации JVM-гарантиями.
