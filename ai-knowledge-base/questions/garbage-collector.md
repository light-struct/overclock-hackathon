# Garbage Collector — Банк вопросов

> **Тема:** Сборщик мусора в Java (Garbage Collector)  
> **Префикс ID:** `gc`  
> **Всего вопросов:** 20 (7 лёгких · 7 средних · 6 сложных)

---

## 🟢 Лёгкие (`easy`)

---

### Q1: Что такое Garbage Collector в Java?

- **ID:** `gc-easy-01`
- **Difficulty:** easy
- **Tags:** GC, основы, память

**Варианты:**
- A) Инструмент для удаления временных файлов на диске
- B) Автоматический механизм управления памятью, освобождающий heap от недостижимых объектов
- C) Планировщик потоков выполнения JVM
- D) Профилировщик использования памяти приложением

**Ответ:** `B`

**Объяснение:** Garbage Collector (GC) — автоматический менеджер памяти JVM. Отслеживает достижимость объектов и освобождает память, занятую объектами на которые нет ссылок. Это устраняет необходимость ручного управления памятью (как в C/C++) и предотвращает большинство утечек памяти.

---

### Q2: Какую область памяти JVM очищает Garbage Collector?

- **ID:** `gc-easy-02`
- **Difficulty:** easy
- **Tags:** heap, память, JVM

**Варианты:**
- A) Stack (стек вызовов)
- B) Heap (куча) — область для хранения объектов
- C) Metaspace — область для метаданных классов
- D) Все области памяти JVM одновременно

**Ответ:** `B`

**Объяснение:** GC работает с Heap — областью памяти где хранятся все объекты Java. Stack очищается автоматически при выходе из метода (локальные переменные). Metaspace (метаданные классов) управляется отдельно. Native memory — вне управления GC.

---

### Q3: Когда объект становится кандидатом для сборки мусора?

- **ID:** `gc-easy-03`
- **Difficulty:** easy
- **Tags:** достижимость, GC-roots

**Варианты:**
- A) После явного вызова System.gc()
- B) Когда на объект больше нет достижимых сильных ссылок
- C) Через 60 секунд после последнего использования объекта
- D) Только когда переменная явно установлена в null

**Ответ:** `B`

**Объяснение:** Объект становится кандидатом для GC когда на него нет достижимых сильных ссылок из GC Roots (локальные переменные в стеке, статические поля, JNI-ссылки). Установка переменной в null — один из способов убрать ссылку, но не единственный и не обязательный.

---

### Q4: Что такое Minor GC?

- **ID:** `gc-easy-04`
- **Difficulty:** easy
- **Tags:** Minor-GC, Young-Generation, поколения

**Варианты:**
- A) Полная сборка всего heap-пространства
- B) Сборка мусора только в области Young Generation (Eden + Survivor)
- C) Принудительная сборка мусора через вызов System.gc()
- D) Инкрементальная сборка небольших объектов

**Ответ:** `B`

**Объяснение:** Minor GC — быстрая сборка в Young Generation (Eden + Survivor spaces S0, S1). Большинство объектов умирают молодыми — Minor GC очищает много памяти быстро. Частые Minor GC — норма. Major/Full GC — более редкая и дорогая операция на всём heap.

---

### Q5: Что такое Stop-The-World (STW) пауза?

- **ID:** `gc-easy-05`
- **Difficulty:** easy
- **Tags:** STW, паузы, производительность

**Варианты:**
- A) Принудительная пауза GC по команде разработчика
- B) Остановка всех потоков приложения на время выполнения определённых фаз GC
- C) Таймаут ожидания завершения GC-цикла
- D) Пауза между Minor GC и Major GC для стабилизации системы

**Ответ:** `B`

**Объяснение:** STW — все потоки приложения замораживаются пока GC выполняет определённые фазы. Это гарантирует консистентность при анализе графа объектов. Длинные STW-паузы = лаги в приложении. Цель современных GC (G1, ZGC, Shenandoah) — минимизировать STW-паузы.

---

### Q6: Что такое Young Generation в heap?

- **ID:** `gc-easy-06`
- **Difficulty:** easy
- **Tags:** Young-Generation, поколения, Eden

**Варианты:**
- A) Область для долгоживущих объектов старше 15 GC-циклов
- B) Область куда помещаются новые объекты; состоит из Eden и двух Survivor spaces
- C) Область для хранения метаданных классов и методов
- D) Зарезервированная область для статических полей

**Ответ:** `B`

**Объяснение:** Young Generation: Eden (новые объекты), Survivor 0, Survivor 1. Новые объекты создаются в Eden. После Minor GC: выжившие перемещаются в Survivor. После нескольких Minor GC выжившие объекты "продвигаются" в Old Generation (tenuring).

---

### Q7: Гарантирует ли System.gc() немедленный запуск GC?

- **ID:** `gc-easy-07`
- **Difficulty:** easy
- **Tags:** System.gc, запуск-GC

**Варианты:**
- A) Да, GC запускается немедленно и синхронно
- B) Нет, это лишь подсказка JVM которую она может проигнорировать
- C) Да, но только Full GC, Minor GC не запускается
- D) System.gc() полностью deprecated и не работает

**Ответ:** `B`

**Объяснение:** `System.gc()` — это hint (подсказка) для JVM, а не команда. JVM может проигнорировать вызов. Опция JVM `-XX:+DisableExplicitGC` полностью игнорирует `System.gc()`. В продакшн коде явный вызов `System.gc()` — антипаттерн. Лучше доверять JVM управлять памятью самостоятельно.

## 🟡 Средние (`medium`)

### Q8: Что такое поколенческая сборка мусора (Generational GC)?

- **ID:** `gc-medium-01`
- **Difficulty:** medium
- **Tags:** generational-GC, поколения, гипотеза

**Варианты:**
- A) GC, работающий параллельно с несколькими независимыми потоками
- B) Разделение heap на поколения (Young/Old) на основе гипотезы: большинство объектов умирают молодыми
- C) GC, запускающийся строго каждые N миллисекунд
- D) Приоритетная сборка мусора по возрасту объектов от старых к новым

**Ответ:** `B`

**Объяснение:** Generational Hypothesis: большинство объектов живут очень короткое время. Поэтому: Young Generation собирается часто и быстро (Minor GC). Old Generation — реже и дольше (Major GC). Это позволяет большинству GC-циклов быть быстрыми, обрабатывая только молодые объекты.

---

### Q9: Что такое Old Generation (Tenured)?

- **ID:** `gc-medium-02`
- **Difficulty:** medium
- **Tags:** Old-Generation, Tenured, Major-GC

**Варианты:**
- A) Область для временных объектов, создаваемых в циклах
- B) Область для объектов переживших несколько Minor GC; собирается реже через Major/Full GC
- C) Зарезервированная системная область JVM
- D) Область heap исключительно для примитивных оболочек

**Ответ:** `B`

**Объяснение:** Old Generation (Tenured): объекты попадают сюда когда переживают определённое число Minor GC (tenuring threshold, по умолчанию 15). Крупные объекты могут сразу создаваться в Old Generation. Full GC — очищает весь heap включая Old Generation, самая дорогая операция.

### Q10: Какой GC используется по умолчанию в Java 9+?

- **ID:** `gc-medium-03`
- **Difficulty:** medium
- **Tags:** G1-GC, JVM-флаги, выбор-GC

**Варианты:**
- A) Serial GC (однопоточный)
- B) Parallel GC (многопоточный throughput-GC)
- C) G1 GC (Garbage First)
- D) ZGC (масштабируемый low-latency GC)

**Ответ:** `C`

**Объяснение:** G1 GC — стал GC по умолчанию с Java 9. Разбивает heap на регионы (~1-32MB), собирает сначала регионы с наибольшим количеством мусора ("Garbage First"). Балансирует throughput и latency. Для Java 8: Parallel GC по умолчанию. Для экстремально низких пауз: ZGC или Shenandoah.

### Q11: Что такое GC Roots?

- **ID:** `gc-medium-04`
- **Difficulty:** medium
- **Tags:** GC-roots, достижимость, mark-phase

**Варианты:**
- A) Первый объект, созданный при запуске JVM
- B) Точки начала трассировки достижимости: стек-фреймы потоков, статические поля, JNI-ссылки
- C) Корневой класс Object из которого наследуются все классы
- D) Метаданные классов в Metaspace

**Ответ:** `B`

**Объяснение:** GC Roots — стартовые точки для обхода графа объектов (Mark phase). Виды GC Roots: локальные переменные и параметры активных методов (стек); статические поля классов; JNI-ссылки из нативного кода; внутренние ссылки JVM. Объект достижим если до него можно добраться по цепочке ссылок от любого GC Root.

### Q12: Что такое WeakReference?

- **ID:** `gc-medium-05`
- **Difficulty:** medium
- **Tags:** WeakReference, ссылки, GC

**Варианты:**
- A) Ссылка на объект, хранящийся в Old Generation
- B) Слабая ссылка — объект может быть собран GC если нет других сильных ссылок на него
- C) Ссылка с автоматической очисткой через метод finalize()
- D) Ссылка для безопасной передачи объектов между потоками

**Ответ:** `B`

**Объяснение:** `WeakReference<T>` — объект доступен пока есть сильные ссылки на него. Как только сильных ссылок нет — GC может собрать объект при следующем цикле. `wr.get()` возвращает объект или null (если собран). Используется в WeakHashMap, кэшах, слушателях событий.

---

### Q13: Чем SoftReference отличается от WeakReference?

- **ID:** `gc-medium-06`
- **Difficulty:** medium
- **Tags:** SoftReference, WeakReference, кэш

**Варианты:**
- A) Они полностью идентичны по поведению
- B) SoftReference очищается только при нехватке памяти (хороша для кэшей); WeakReference очищается при первом же GC
- C) WeakReference очищается медленнее чем SoftReference
- D) SoftReference работает только для строк и примитивных обёрток

**Ответ:** `B`

**Объяснение:** `SoftReference`: объект живёт до тех пор пока JVM не начнёт испытывать нехватку памяти. Идеально для кэшей — JVM сама решает когда очистить. `WeakReference`: объект собирается при ближайшем GC если нет сильных ссылок. Для ассоциаций без владения (WeakHashMap). `PhantomReference`: самая слабая, всегда возвращает null.

### Q14: Что такое Memory Leak в Java?

- **ID:** `gc-medium-07`
- **Difficulty:** medium
- **Tags:** memory-leak, достижимость

**Варианты:**
- A) Ошибка JVM при выделении памяти для нового объекта
- B) Ситуация когда объекты остаются достижимыми (есть ссылки) но больше не используются — GC не может их собрать
- C) Переполнение стека вызовов при рекурсии
- D) Утечка памяти только в нативном (native) коде

**Ответ:** `B`

**Объяснение:** В Java GC собирает только недостижимые объекты. Если код держит ссылки на объекты которые больше не нужны — это memory leak. Примеры: static коллекция накапливающая данные; Event listeners без отписки; ThreadLocal без remove(); кэши без ограничения размера.

### Q15: Как работает G1 GC и что такое регионы (regions)?

- **ID:** `gc-hard-01`
- **Difficulty:** hard
- **Tags:** G1-GC, регионы, алгоритм

**Варианты:**
- A) G1 делит heap на два больших блока: Young и Old с фиксированным размером
- B) G1 делит heap на множество равных регионов (~1-32MB); каждый регион может быть Eden, Survivor, Old или Humongous; GC собирает регионы с наибольшим количеством мусора сначала
- C) G1 работает только с одним потоком для минимальных накладных расходов
- D) G1 не использует STW-паузы вообще

**Ответ:** `B`

**Объяснение:** G1 (Garbage First): heap делится на ~2000 равных регионов. Каждый регион помечается как Eden, Survivor, Old или Humongous (для очень больших объектов). G1 оценивает количество мусора в каждом регионе и в первую очередь собирает регионы с наибольшим выигрышем ("Garbage First"). Цель: предсказуемые паузы с настраиваемым целевым временем (`-XX:MaxGCPauseMillis`).

### Q16: Что такое ZGC и какова его основная цель?

- **ID:** `gc-hard-02`
- **Difficulty:** hard
- **Tags:** ZGC, low-latency, concurrent-GC

**Варианты:**
- A) Zero-cost GC — работает без накладных расходов на CPU
- B) Масштабируемый конкурентный GC с паузами менее 1-10ms независимо от размера heap
- C) GC специально разработанный как улучшение G1 для малых heap
- D) Сборщик мусора для Kotlin и Groovy приложений на JVM

**Ответ:** `B`

**Объяснение:** ZGC (Java 11+, production ready Java 15+): цель — паузы < 1-10ms даже для heap в несколько терабайт. Работает конкурентно: большинство работы выполняется параллельно с потоками приложения. Использует Load Barriers и Colored Pointers для конкурентного перемещения объектов. Shenandoah GC — аналогичная цель, но другой алгоритм.

### Q17: Что такое finalize() и почему он deprecated с Java 9?

- **ID:** `gc-hard-03`
- **Difficulty:** hard
- **Tags:** finalize, deprecated, очистка-ресурсов

**Варианты:**
- A) Метод, гарантированно вызываемый GC перед удалением объекта — аналог деструктора C++
- B) Устаревший метод в Object, который может быть вызван GC перед очисткой; deprecated из-за непредсказуемости, overhead и проблем с resurrection
- C) Метод для принудительного запуска GC из кода объекта
- D) Метод для регистрации объекта как не подлежащего сборке мусора

**Ответ:** `B`

**Объяснение:** Проблемы finalize(): 1) Непредсказуемость — JVM не гарантирует когда и будет ли вызван. 2) Производительность — объекты с finalize() проходят два GC-цикла. 3) Object resurrection — можно "воскресить" объект, сохранив ссылку на него в finalize(). 4) Deadlocks если finalize() блокируется. Замены: try-with-resources, Cleaner API (Java 9+), PhantomReference.

### Q18: Что такое card table и write barrier в GC?

- **ID:** `gc-hard-04`
- **Difficulty:** hard
- **Tags:** card-table, write-barrier, generational-GC

**Варианты:**
- A) Таблица для хранения метаданных о размере объектов в heap
- B) Card table — структура данных для отслеживания ссылок из Old в Young; write barrier — код, обновляющий card table при изменении ссылки
- C) Барьер для синхронизации потоков во время GC-паузы
- D) Инструмент профилирования для обнаружения memory leaks

**Ответ:** `B`

**Объяснение:** Проблема Minor GC: нужно найти ссылки из Old Generation на Young Generation (без этого нельзя знать что в Young достижимо). Card Table: heap делится на "карточки" по 512 байт. Write Barrier: при каждом изменении ссылки (`object.field = newValue`) JVM помечает карточку как "dirty". Minor GC проверяет только dirty cards в Old Generation.

### Q19: Как обнаружить и исправить memory leak в Java-приложении?

- **ID:** `gc-hard-05`
- **Difficulty:** hard
- **Tags:** memory-leak, профилирование, инструменты

**Варианты:**
- A) Memory leak в Java невозможен так как GC управляет памятью автоматически
- B) Использовать heap profiler (VisualVM, JProfiler, Eclipse MAT) для анализа heap dump; искать классы с неожиданно большим количеством экземпляров; проверить static коллекции, listeners, ThreadLocal
- C) Перезапускать JVM каждые несколько часов — это решает проблему
- D) Добавить больше RAM серверу

**Ответ:** `B`

**Объяснение:** Диагностика: 1) Мониторинг: heap растёт и не очищается после Full GC. 2) Heap dump: `-XX:+HeapDumpOnOutOfMemoryError`. 3) Анализ в Eclipse MAT или VisualVM: найти Retained Heap (что удерживается). 4) Типичные источники: static Map/List без очистки; EventListeners без remove; ThreadLocal без remove; inner классы удерживающие outer; Unclosed Streams/Connections.

### Q20: Как работает Escape Analysis и как он влияет на выделение памяти?

- **ID:** `gc-hard-06`
- **Difficulty:** hard
- **Tags:** Escape-Analysis, JIT, оптимизация, стек

**Варианты:**
- A) Анализ утечек памяти профилировщиком во время работы приложения
- B) JVM-оптимизация JIT-компилятора: если объект не "вырывается" за пределы метода — JVM может выделить его на стеке вместо heap, что ускоряет работу и снижает нагрузку на GC
- C) Анализ исключений и их путей в стеке вызовов
- D) Оптимизация строковых литералов через String Pool

**Ответ:** `B`

**Объяснение:** Escape Analysis — JIT-оптимизация. JVM анализирует: "вырывается" ли объект за пределы метода (передаётся в другой метод, сохраняется в поле, возвращается)? Если нет — объект "не вырывается" (no escape). Оптимизации: Stack Allocation — объект создаётся на стеке (O(1) выделение, автоматически очищается). Scalar Replacement — объект заменяется скалярными полями без создания объекта. Это снижает давление на GC значительно.
